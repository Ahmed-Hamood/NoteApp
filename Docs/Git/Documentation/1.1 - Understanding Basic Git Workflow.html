1.1 - Understanding Basic Git Workflow

Now our project directory is using Git, inside our project we have Git Repository which is a hidden sub-directory in our project directory. In daily basis tasks we add or modify files in our project directory, sometime our project state we want to record/save these changes or we can say commit those changes into our repository

Creating a commit its like taking a snapshot of our project or saving the current project state. There is a special area that comes between before and after committing our project files, this special intermediate area called "Staging Area" or "Index", this stage comes before commit.

The Staging Area is place where we store our snapshot before we commit them. When we make changes like adding or modifying file, we add these files to the staging area, here we review our changes if everything is good then we make a commit or snapshot record that will get our files permanently stored in our repository.

As we say the staging area allows us to review our work before recording a snapshot, if some changes shouldn't be recorded as part of the next snapshot then we can easily un-stage them and commit them as a part of another snapshot as deleted/removed simple as that.

Each commit has a unique identifier that gets generated by Git like revision number. every commit contain information like.

  • ID
  • Message
  • Date/time
  • Author
  • Completed snapshot at the time it was created

How Git stores data, Git internal functionality or implementation Compress our content and doesn't store duplicate content. Each commit contain a complete snapshot of our project, and this allow us to quickly get back to a previous state

- Real Example

Here's an example, let's say er have a couple of files file1 and file2, and they are ready to be in committed/recorded state, before that we have to add them to staging area state before committing them, to add those file into the staging area we use the following command. 

  git add file1 file2

Now our files in the staging area, we review these files everything is good, so we decider to commit these files by using the commit command to permanently store as a snapshot into the repository we use the following command 

  git commit -m "initial commit"

We supply a meaningful message to indicate what the snapshot represents, this only is useful to have a useful history. Each commit or snapshot that you save has a message that explains the state of the project at that point in time

When you commit our snapshot, a copy of our files remain in the staging area. Let's say we fixed a bug so we make some changes to file1, as we know currently we have an old version of file in the staging area because we haven't staged the changes yet

Once again we use the add command to stage the changes, now we have the updated file1 in the staging area which is the same file in the working directory.

So, lets make a commit to record this state, now we have two commits in our repository, the second commit message it's describes the bug that we just fixed.

Now, we realize that we no longer need file2 because it contains unused code, so we delete it from our working directory, but still file2 in the staging area.

So, again we should use the add file2 command to stage this change in this case the deletion occur, although Git know that file2 is deleted, so it will deleted also from the staging area ready for next commit snapshot. Now we can commit our changes.
